Agent webhooks are HTTP callbacks fired by the platform when an agent run completes or an approval changes state. Use them to forward agent activity into your own systems - moderation dashboards, audit logs, Slack channels, escalation tools.

Configured under the **Webhooks** tab on the [AI Agents page](https://fastcomments.com/auth/my-account/ai-agents).

### Что доставляется

Четыре типа событий:

- **`trigger.succeeded`** - выполнение агента успешно завершилось.
- **`trigger.failed`** - выполнение агента завершилось с ошибкой.
- **`approval.requested`** - действие поставлено в очередь на ручное утверждение.
- **`approval.decided`** - утверждение было одобрено, отклонено или выполнение завершилось ошибкой.

Смотрите [Webhook Events](#webhook-events), чтобы узнать, какие события и когда срабатывают, и [Webhook Payloads](#webhook-payloads) для схемы каждого полезной нагрузки.

### Что не доставляется

- **Per-tool-action webhooks.** Запуск, который вызывает `pin_comment`, не генерирует отдельный вебхук для закрепления — действие включено в полезную нагрузку `trigger.succeeded` выполнения. Если вам нужна доставка по каждому действию отдельно, разбирайте массив `actions` в полезной нагрузке триггера.
- **Dropped triggers.** Триггер, который не запускается (превышен бюджет, неверная область), не генерирует вебхук. Отказы видны только на странице [Analytics page](#analytics-page).
- **Replay-produced triggers.** Тестовые запуски не отправляют вебхуки. Смотрите [Test Runs (Replays)](#test-runs-replays).

### Конфигурация

Для каждого вебхука вы задаете:

- **URL** - HTTPS-эндпоинт для POST-запросов.
- **Domain** - домен комментариев, к которому применяется этот вебхук (у вашего тенанта может быть несколько доменов). `*` соответствует всем доменам; `*.example.com` — это глоб (подстановочный шаблон); точное доменное имя совпадает только с ним самим.
- **Events** - на какие из четырёх типов событий подписываться.
- **Agents** - пустое значение означает «все агенты», либо список конкретных ID агентов для фильтрации.
- **Method** - POST или PUT (по умолчанию POST).
- **No-retry status codes** - HTTP-коды ответа, которые считаются окончательной ошибкой и не будут повторяться (например, 410 Gone, 422 Unprocessable). Смотрите [Webhook Retries](#webhook-retries).

Несколько вебхуков могут подписываться на одно и то же событие — каждый помещается в очередь независимо и доставляется на собственный URL.

### Соответствие по домену

Конкретное событие доставляется во **все вебхуки, поле `domain` которых соответствует домену комментария**. Сопоставление использует простой глоб:

- Точное совпадение: `example.com` соответствует только `example.com`.
- Подстановочный символ `*`: `*` соответствует каждому домену.
- Глоб поддоменов: `*.example.com` соответствует `blog.example.com`, `forum.example.com`, но не `example.com`.

Домен в полезной нагрузке — это первый ненулевой результат из: `domain` комментария, URL, распарсенного в соответствии с конфигурацией доменов вашего тенанта, или поиска `Page` по `urlId`.

### Фильтрация по агенту

Поле **Agents** позволяет вебхуку подписываться только на определённых агентов. Пустое значение означает «все агенты». Если указано ненулевое значение, вебхук срабатывает только для агентов из списка.

Это полезно, когда у вас есть один вебхук для модерации и другой для вовлечения, которые направляются в разные downstream-системы.

### Test send

В UI конфигурации вебхука есть кнопка **Test send**, которая отправляет фейковую полезную нагрузку на URL, чтобы вы могли проверить подключение, подпись и код ответа вашего эндпоинта без ожидания реального события.

### Журналы доставки

Каждая доставка (и каждая попытка повторной отправки) попадает на страницу [Webhook Delivery Logs](#webhook-logs), где вы можете увидеть, что произошло при передаче.

### Подпись

Каждый вебхук подписывается HMAC-SHA256 с использованием секретного ключа API вашего тенанта. Смотрите [Webhook Signing](#webhook-signing).

### Доступность

Вебхуки агентов требуют действующей оплаты на тенанте. Они используют ту же инфраструктуру подписи/секрета, что и ваши существующие вебхуки комментариев — если вы уже интегрировали вебхуки комментариев (см. [Webhooks guide](/guide-webhooks.html)), интеграция вебхуков агентов имеет ту же структуру с новыми типами событий.
