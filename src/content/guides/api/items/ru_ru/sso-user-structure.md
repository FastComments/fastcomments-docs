FastComments предоставляет простое в использовании решение SSO. Обновление информации пользователя с помощью интеграции на основе HMAC
так же просто, как загрузка пользователем страницы с обновлённым payload.

Однако может потребоваться управлять пользователем вне этого потока, чтобы улучшить согласованность вашего приложения.

SSO User API предоставляет способ выполнять CRUD над объектами, которые мы называем SSOUsers. Эти объекты отличаются от обычных Users и
хранятся отдельно ради безопасности типов.

Структура объекта SSOUser выглядит следующим образом:

[inline-code-attrs-start title = 'Структура SSOUser'; type = 'typescript'; isFunctional = false; inline-code-attrs-end]
[inline-code-start]
interface SSOUser {
    id: string
    username: string
    email?: string
    websiteUrl?: string
    signUpDate: number
    createdFromUrlId?: string
    loginCount?: number
    avatarSrc?: string
    optedInNotifications?: boolean
    optedInSubscriptionNotifications?: boolean
    displayLabel?: string
    displayName?: string
    isAccountOwner?: boolean // Права администратора - SSO-пользователи с этим флагом учитываются как SSO Admins (отдельно от обычных SSO пользователей)
    isAdminAdmin?: boolean // Права администратора - SSO-пользователи с этим флагом учитываются как SSO Admins (отдельно от обычных SSO пользователей)
    isCommentModeratorAdmin?: boolean // Права модератора - SSO-пользователи с этим флагом учитываются как SSO Moderators (отдельно от обычных SSO пользователей)
    /** Если null, Контроль доступа не будет применён к пользователю. Если пустой список, этот пользователь не сможет видеть какие-либо страницы или @упоминать других пользователей. **/
    groupIds?: string[] | null
    createdFromSimpleSSO?: boolean
    /** Не позволять другим пользователям видеть активность этого пользователя, включая комментарии, на его профиле. По умолчанию true для обеспечения защищённых профилей. **/
    isProfileActivityPrivate?: boolean
    /** Не позволять другим пользователям оставлять комментарии на профиле пользователя или видеть существующие комментарии профиля. По умолчанию false. **/
    isProfileCommentsPrivate?: boolean
    /** Не позволять другим пользователям отправлять этому пользователю личные сообщения. По умолчанию false. **/
    isProfileDMDisabled?: boolean
    karma?: number
    /** Необязательная конфигурация бейджей пользователя. **/
    badgeConfig?: {
        /** Массив ID бейджей, назначаемых пользователю. Ограничение — 30 бейджей. Порядок сохраняется. Это глобальные бейджи, видимые на всех страницах. **/
        badgeIds: string[]
        /** Массив ID бейджей, привязанных к текущей странице (urlId). Эти бейджи отображаются только на странице, где они были назначены. **/
        pageBadgeIds?: string[]
        /** Если true, заменяет все существующие отображаемые бейджи на предоставленные. Глобальные и привязанные к странице бейджи переопределяются независимо. Если false, добавляет к существующим бейджам. **/
        override?: boolean
        /** Если true, обновляет свойства отображения бейджей из конфигурации тенанта. **/
        update?: boolean
    }
}

### Billing for SSO Users

SSO users are billed differently based on their permission flags:

- **Regular SSO Users**: Пользователи без прав администратора или модератора учитываются как обычные SSO пользователи
- **SSO Admins**: Пользователи с флагами `isAccountOwner` или `isAdminAdmin` учитываются отдельно как SSO Admins (та же ставка, что и для обычных администраторов tenant)
- **SSO Moderators**: Пользователи с флагом `isCommentModeratorAdmin` учитываются отдельно как SSO Moderators (та же ставка, что и для обычных модераторов)

**Важно**: Чтобы предотвратить двойное выставление счетов, система автоматически дедуплицирует SSO пользователей по сравнению с обычными пользователями tenant и модераторами по адресу электронной почты. Если у SSO пользователя тот же email, что и у обычного пользователя tenant или модератора, за него не выставят счёт дважды.

### Access Control

Пользователей можно разбивать на группы. Для этого и служит поле `groupIds`, оно опционально.

### @Mentions

По умолчанию при вводе символа `@` поиск `@mentions` будет использовать `username` для поиска других sso пользователей. Если используется `displayName`, то результаты, совпадающие с `username`, будут игнорироваться при наличии совпадения по `displayName`, и результаты поиска `@mention` будут использовать `displayName`.

### Subscriptions

С помощью FastComments пользователи могут подписываться на страницу, нажимая значок колокольчика в виджете комментариев и выбирая Subscribe.

Для обычного пользователя мы отправляем уведомления по электронной почте в соответствии с его настройками уведомлений.

Для SSO пользователей мы разделили это для обратной совместимости. Дополнительные уведомления о подписках будут отправляться пользователям только если вы установите `optedInSubscriptionNotifications` в `true`.

### Badges

Вы можете назначать бейджи SSO пользователям с помощью свойства `badgeConfig`. Бейджи — это визуальные индикаторы, которые отображаются рядом с именем пользователя в комментариях.

- `badgeIds` - Массив ID бейджей, назначаемых пользователю. Это глобальные бейджи, видимые на всех страницах. Должны быть действительными ID бейджей, созданных в вашей учетной записи FastComments. Ограничение — 30 бейджей.
- `pageBadgeIds` - Необязательный массив ID бейджей, привязанных к текущей странице (`urlId`). Эти бейджи отображаются только на странице, где они были назначены. Разные страницы могут иметь разные привязанные к странице бейджи для одного и того же пользователя.
- `override` - Если true, все существующие отображаемые бейджи будут заменены на предоставленные. Глобальные и привязанные к странице бейджи переопределяются независимо — переопределение глобальных бейджей не влияет на привязанные к странице, и наоборот. Если false или опущено, предоставленные бейджи будут добавлены к существующим.
- `update` - Если true, свойства отображения бейджей будут обновляться из конфигурации тенанта при каждом входе пользователя.
